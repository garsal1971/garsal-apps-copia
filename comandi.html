Aggiungi colonne mancanti a cm_notification_rules
La tabella esistente fu creata con lo schema base (solo app, entity_id, rule_type, fire_at, payload, active).
La funzionalità "Notifica al Volo" usa colonne nuove: entity_title, entity_type, channel, enabled, reminder_presets, created_at.
Esegui questo script nel SQL Editor del progetto Supabase PROD.
SQL PROD cm_notification_rules
⚠ Esegui nel progetto Supabase PROD: jajlmmdsjlvzgcxiiypk.supabase.co
Le istruzioni usano IF NOT EXISTS / IF NOT EXISTS (colonna) — sono sicure da rieseguire. 
-- ══════════════════════════════════════════════════════════════
-- garsal-apps — Migrazione cm_notification_rules (PROD)
-- Aggiunge le colonne usate da "Notifica al Volo" (app-launcher)
-- Sicuro da eseguire più volte (IF NOT EXISTS)
-- ══════════════════════════════════════════════════════════════

-- 1. entity_title: testo libero della notifica
ALTER TABLE cm_notification_rules
  ADD COLUMN IF NOT EXISTS entity_title TEXT;

-- 2. entity_type: tipologia ('once', 'recurring', ecc.)
ALTER TABLE cm_notification_rules
  ADD COLUMN IF NOT EXISTS entity_type TEXT;

-- 3. channel: canale di invio ('telegram', 'push', 'email')
ALTER TABLE cm_notification_rules
  ADD COLUMN IF NOT EXISTS channel TEXT DEFAULT 'telegram';

-- 4. enabled: regola attiva o disabilitata
--    (esisteva già come "active" — aggiungiamo "enabled" in parallelo)
ALTER TABLE cm_notification_rules
  ADD COLUMN IF NOT EXISTS enabled BOOLEAN DEFAULT true;

-- 5. reminder_presets: JSONB con fire_at, telegram_snooze_button, ecc.
--    (esisteva già come "payload" — aggiungiamo "reminder_presets")
ALTER TABLE cm_notification_rules
  ADD COLUMN IF NOT EXISTS reminder_presets JSONB;

-- 6. created_at: timestamp di creazione (usato per ordinamento)
ALTER TABLE cm_notification_rules
  ADD COLUMN IF NOT EXISTS created_at TIMESTAMPTZ DEFAULT now();

-- Verifica risultato
SELECT column_name, data_type, column_default
FROM information_schema.columns
WHERE table_name = 'cm_notification_rules'
ORDER BY ordinal_position;
